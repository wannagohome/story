# Story — AI 구동 멀티플레이어 텍스트 RPG 설계 문서

**작성일:** 2026-03-01

---

## 한 줄 정의

룸 코드 하나로 모인 친구들이, AI가 즉석에서 만든 세계에서 각자의 역할을 받고 채팅과 이동으로 스토리를 풀어나가는 터미널 소셜 RPG.

---

## 타겟 유저 (우선순위 순)

1. **소셜 게이머 (친구 그룹)** — 핵심. 저녁 모임, 술자리, 파티에서 함께 즐기는 형태
2. 개발자 / 기술 친화적 유저
3. TRPG / 보드게임 팬
4. 일반 캐주얼 게이머

---

## 핵심 설계 원칙

### 1. 진입 장벽 제로
별도 설치 없이 `npx` 한 줄로 실행. 룸 코드 하나로 친구들이 즉시 모임. 세계관, 역할, 스토리 모두 AI가 자동 생성하므로 플레이어는 아무것도 준비하지 않아도 된다.

### 2. 방(Room)이 곧 프라이버시 단위
별도의 밀담/귓속말 기능을 두지 않는다. 밀담이 필요하면 조용한 방으로 이동하면 된다. 이 구조가 더 자연스럽고, 이동 자체가 의심의 단서가 되는 게임 메커닉을 만들어낸다.

### 3. GM과 NPC는 스토리에 따라 유동적
모든 게임에 GM이 있거나 NPC가 있는 것이 아니다. AI가 세계를 생성할 때 이 스토리에 GM이 필요한지, NPC가 있어야 하는지를 함께 결정한다. 플레이어들이 의식하지 않아도 자연스럽게 존재하거나 존재하지 않는다.

### 4. 정보 비대칭이 게임의 동력
공통 기반(공개 정보)이 없으면 대화가 시작되지 않고, 비밀(비공개 정보)이 없으면 긴장감이 없다. 두 레이어의 균형이 플레이어들이 서로 대화하고, 이동하고, 동맹을 맺고, 의심하게 만드는 핵심 엔진이다.

### 5. 재미와 몰입이 모든 것의 기준
스토리, 설정, 게임 구조 설계의 최우선 기준은 게임의 재미와 몰입도다. AI는 항상 이 방향으로 세계를 만들고 이야기를 이끌어야 한다.

## 배포 방식

```
npx story host      # 호스트로 게임 생성 및 서버 실행
npx story join <룸코드>  # 참가자로 입장
```

npm 패키지로 배포. 별도 설치 불필요.

## 게임 플로우

### 세션 시작

```
1. 호스트가 `npx story host` 실행
   → AI가 세계 생성 (장르/분위기/게임 구조 자동 결정)
   → 룸 코드 발급 (예: WOLF-7423)

2. 친구들에게 룸 코드 공유
   → 참가자들이 `npx story join WOLF-7423` 으로 입장

3. 모든 플레이어가 모이면 AI 브리핑 시작
   → 공개 정보 전달 (사건 개요, 세계관, 등장인물 명단)
   → 각 플레이어에게 역할 & 비공개 정보 개별 전달 (다른 플레이어 화면에는 보이지 않음)

4. 게임 시작
```

### 게임 진행 중 가능한 행동

| 행동 | 설명 | 보이는 대상 |
|------|------|-------------|
| 글로벌 채팅 | 공개 발언 | 전체 플레이어 |
| 같은 방 대화 | 현재 위치 방의 대화 | 같은 방 플레이어만 |
| 이동 | 맵의 다른 방으로 이동 | 이동 사실은 전체 공개 |
| NPC 대화 | 같은 방 NPC와 대화 | 같은 방 플레이어 모두 |
| `/examine` | 방 안의 물건/환경 탐색 | 같은 방 플레이어 모두 |
| `/do` | 행동 서술 (롤플레이) | 같은 방 플레이어 모두 |
| `/end` | 게임 종료 발의 (과반수 동의 시 종료) | 전체 플레이어 |

### 세션 종료

AI가 세계 생성 시 스토리에 맞는 종료 조건을 자유롭게 설계한다. 투표, 합의, 특정 이벤트 달성, AI 판단 등 어떤 형태든 가능하다.

| 종료 방식 | 설명 |
|-----------|------|
| AI가 설계한 종료 조건 달성 | 투표, 합의, 이벤트 달성 등 스토리에 맞게 AI가 설계 |
| 플레이어 주도 종료 (`/end`) | 아무 플레이어가 종료를 발의하고, 과반수 동의 시 종료 |
| 시간 초과 시 AI가 강제로 결말 서술 | 항상 fallback으로 포함 |

### 엔딩

단순한 정답 공개가 아니다. AI가 **각 플레이어의 행동을 돌아보며** 개인화된 결말을 서술한다.

```
[게임 종료]

범인은 Bob이었습니다.

Alice는 끝까지 Carol을 의심했지만, 마지막 순간
Bob의 거짓말을 간파했습니다.
Carol은 자신의 비밀을 끝내 지켜냈습니다.
Bob은 거의 성공할 뻔 했으나, 서재에서의 실수가
결정적이었습니다.
```

## 게임 구조

**사전 정의된 모드 분류(머더 미스터리, 협력 추리 등)를 두지 않는다.** AI가 세계 생성 시 스토리에 맞는 게임 구조를 자유롭게 설계한다. **한 판의 게임은 10~30분 내에 완결되도록 설계한다.** 세션 길이도 AI가 플레이어 수와 진행 속도에 따라 유동적으로 조절하되, 최대 30분을 넘지 않는다.

AI가 세계와 함께 설계하는 것들:
- **핵심 갈등** — 무엇이 플레이어를 움직이게 하는가
- **진행 방식** — 어떻게 게임이 전개되는가
- **공통 목표** — 있을 수도, 없을 수도 있음
- **종료 조건** — 어떤 상태가 되면 게임이 끝나는가 (투표, 합의, 이벤트, AI 판단 등 제한 없음)
- **승리 판정** — 어떻게 결과를 평가하는가

이 구조 덕분에 AI는 머더 미스터리, 협력 탈출, 정치 드라마, 생존, 경매, 또는 아직 이름 붙이지 못한 새로운 형태의 게임까지 자유롭게 만들어낼 수 있다.

### 게임 구조 예시 (AI가 만들 수 있는 것들)

| 핵심 갈등 | 진행 방식 | 종료 조건 |
|-----------|-----------|-----------|
| 6인의 용의자 중 범인 색출 | 단서 수집, 알리바이 검증, 대화 | 투표로 범인 지목 |
| 침몰하는 우주선에서 탈출 | 퍼즐 해결, NPC 설득, 자원 분배 | 탈출 코드 입력 |
| 왕위 계승 전쟁 | 동맹, 배신, 투표, 증거 수집 | 최다 지지 획득 |
| 졸업식 전날 밤의 비밀 | 관계 탐색, 감정 고백, 비밀 유지 | AI가 감정적 절정 판단 |
| 제한된 식량을 둔 생존 | 자원 거래, 협력/배신 선택 | 특정 라운드 생존 |

## 맵 시스템

맵은 **방(채팅방)들의 네트워크**다. 방이 곧 프라이버시 단위이므로 별도의 밀담 기능은 없다. 밀담을 원하면 조용한 방으로 함께 이동하면 된다.

```
        [2층 복도]
           │
[2층 서재] ─┤─ [2층 침실 A] ─ [2층 침실 B]
           │
        [계단]
           │
[주방] ─ [1층 거실] ─ [서재]
           │
        [지하실]
```

- **공개 공간** (거실, 복도): 많은 플레이어가 모이는 허브. 정보 교환과 공개 토론의 장
- **밀실** (침실, 서재): 소수가 모여 은밀한 대화를 나누는 공간. 이동 자체가 의심의 단서가 됨
- **NPC 상주 공간**: NPC가 특정 방에 배치되어 대화 가능

맵 구조는 스토리/배경에 따라 AI가 생성한다 (저택, 우주선, 작은 마을, 학교 등).

**방의 수는 플레이어가 여러 그룹으로 나뉘어 동시에 밀담을 나눌 수 있을 만큼 충분해야 한다.** 기준: 플레이어 수의 절반 이상이 각자 다른 방에 있어도 빈 방이 남을 정도. 예: 6인 게임이면 최소 5~7개 방. 맵이 너무 좁으면 밀담 자체가 불가능해져 게임 메커닉이 무너진다.

### 위치 정보 공개 범위

| 정보 | 공개 대상 |
|------|-----------|
| 내 위치 | 전체 공개 |
| 다른 플레이어 위치 | 전체 공개 |
| 같은 방 대화 내용 | 같은 방 플레이어만 |
| NPC와의 대화 내용 | 같은 방 플레이어 모두 |

누가 어디 있는지는 모두 알 수 있지만, 무슨 말을 했는지는 같은 방에 있어야 알 수 있다.

## AI 역할

### 1. 세계 생성 (게임 시작 시 1회)

**생성 철학: 상상력 최대, 흥미 최우선**

스토리 생성 프롬프트는 의도적으로 모호하고 열려있게 설계한다. AI가 장르, 배경, 사건, 분위기를 최대한 자유롭게 상상해서 만들도록 유도한다. "재미있고 몰입되는 스토리"가 유일한 기준. 개연성보다 흥미가 우선이다.

**멀티 모델 AI 오케스트레이션:** 세계 생성은 단일 AI가 아닌, 서로 다른 성향의 여러 AI 모델이 역할을 나누어 협업하는 "작가실" 구조로 실행된다. 아이디어를 쏟아내는 작가들, 이를 하나의 세계로 통합하는 총괄 작가, 구조적 결함을 잡아내는 검수자가 파이프라인으로 연결된다. 이를 통해 스토리의 독창성, 기믹의 다양성, NPC의 입체적 설계를 확보한다.

생성 항목:
- 배경 세계관, 사건, 분위기
- **게임 구조** (핵심 갈등, 진행 방식, 종료 조건, 승리 판정 — 사전 정의된 모드 없이 자유 설계)
- 맵 구조 및 각 공간의 특성
- 등장인물 (플레이어 역할 + NPC)
- 복선, 단서, 기믹 (숨겨진 장치들)
- 각 플레이어에게 역할 + 개인 목표(1개 이상) 배정
- 공개/반공개/비공개 정보 레이어 구성 및 분배
- GM 필요 여부, NPC 필요 여부 결정

생성된 스토리, 맵, 기믹 등 모든 데이터는 저장된다. 게임 종료 후 플레이어 피드백과 함께 아카이브.

**출력 형식은 타이트하게 구조화된 JSON으로 고정한다.** AI 출력이 자유 형식이면 UI가 깨진다. 모든 AI 응답은 미리 정의된 스키마를 따라야 하며, 해석의 여지 없이 파싱 가능해야 한다.

### 1-1. 스토리 검증 에이전트 (세계 생성 직후 자동 실행)

세계 생성 직후 별도 검증 에이전트가 실행된다. 이 에이전트는 창의성이나 개연성을 판단하지 않는다. **구조적 결함만 검사한다.**

검사 항목:
- 앞뒤가 맞지 않거나 모순되는 설정이 있는가
- 복선이 회수될 수 있는 경로가 실제로 존재하는가
- 단서가 맵 어딘가에 실제로 배치되어 있는가
- NPC가 알고 있다고 설정된 정보가 실제로 NPC에게 부여되어 있는가
- 맵 구성과 스토리 설정 간 불일치가 있는가
- **종료 조건이 달성 가능한 경로가 존재하는가**
- **모든 플레이어에게 의미 있는 행동 경로가 있는가**
- **종료 판정 기준이 명확하고 평가 가능한가**

결함 발견 시 해당 부분만 재생성 → 재검증. 통과하면 게임 시작.

### 2. GM (스토리에 따라 있을 수도, 없을 수도 있음)

AI가 세계를 생성할 때 이 스토리에 GM이 필요한지 함께 결정한다. 외부 개입이 게임의 긴장감이나 진행에 필수적인 스토리에는 GM이 있고, 플레이어 간 상호작용만으로 충분한 스토리에는 GM이 없다.

GM이 있을 때 하는 일:
- 플레이어 행동에 반응해 스토리 이벤트 발생
- 게임이 지루해지면 긴장감을 높이는 사건 투입
- 결말을 향해 스토리가 자연스럽게 수렴하도록 조율

### 3. NPC (스토리에 따라 있을 수도, 없을 수도 있음)

AI가 세계를 생성할 때 NPC가 필요한지 함께 결정한다. 정보 전달자, 기믹 보유자, 분위기 조성자 등 스토리에 NPC가 필요한 역할이 있으면 생성하고, 플레이어만으로 충분하면 생성하지 않는다.

NPC가 있을 때:
- 각 NPC는 독립적인 퍼소나와 보유 정보를 가짐
- 플레이어가 말을 걸면 퍼소나를 유지하며 응답
- 어떤 정보를 줄지, 숨길지는 NPC의 성격과 상황에 따라 결정

**NPC 상호작용은 대화에 한정되지 않는다.** 스토리 설계에 따라 다음이 가능하다:
- 아이템/물건을 플레이어에게 주거나 받음
- 특정 조건이 충족되면 증표나 증거물을 공개 (다음 단계 진행 트리거)
- 특정 행동이나 선택에 반응해 스토리 분기 발생

UI는 이 모든 상호작용 유형을 처리할 수 있도록 설계한다. 단, 모든 상호작용의 결과는 타이트하게 구조화된 이벤트 타입으로 정의되어 있어야 한다 (자유 형식 출력 금지).

**NPC 예시:**
```
[집사 레이몬드]
위치: 1층 거실
퍼소나: 충직하지만 뭔가 숨기는 듯한 태도
아는 것: 피해자가 어젯밤 누군가와 다퉜다는 것
숨기는 것: 본인이 그 자리에 있었다는 것
행동 원칙: 직접 묻지 않으면 먼저 말하지 않는다
기믹: 피해자의 열쇠를 소지 중. 신뢰를 충분히 얻으면 건넴
```

## 플레이어 역할 & 정보 시스템

### 역할 배정 예시

게임 시작 시 AI가 각 플레이어에게 비공개로 전달한다.

```
[당신의 역할]
이름: 박사 에드워드 크레인
배경: 피해자의 오랜 친구이자 라이벌 학자
개인 목표: 피해자가 훔쳐간 연구 노트를 되찾아라
비밀: 어젯밤 11시, 당신은 서재에 있었다
```

### 개인 목표 시스템

**모든 플레이어에게 게임의 공통 목표와 별개로 고유한 개인 목표가 부여된다.** 개인 목표는 1개 이상이며, 게임 구조와 스토리 성격에 따라 성격이 달라진다.

| 목표 유형 | 예시 |
|-----------|------|
| 추리 / 수사 | 범인을 밝혀라 / 증거를 인멸하라 / 특정 인물을 보호하라 |
| 관계 / 감정 | 누군가의 마음을 얻어라 / 비밀을 끝까지 지켜라 |
| 생존 / 자원 | 출구 코드를 찾아라 / 모두를 살려서 탈출하라 |
| 정치 / 전략 | 왕위 계승권을 확보하라 / 동맹을 3명 이상 모아라 |
| 복합 | 범인도 잡고, 동시에 특정 인물도 보호하라 |

**개인 목표 간 긴장 관계가 핵심이다.** 개인 목표가 때로는 공통 목표나 다른 플레이어의 목표와 충돌할 수 있다. 예: 범인을 알지만 개인적 이유로 감싸야 하는 상황. 이 긴장이 플레이어 간 대화, 이동, 동맹, 배신을 만들어낸다.

엔딩에서 공통 목표의 결과와 각 플레이어의 개인 목표 달성 여부가 개별적으로 평가되고 서술된다.

### 정보 레이어 구조

모든 정보가 비공개이면 대화가 시작되지 않는다. 공통 기반(공개 정보)이 있어야 플레이어들이 공유된 세계에서 출발할 수 있다.

| 레이어 | 공개 범위 | 내용 |
|--------|-----------|------|
| **공개** | 모든 플레이어 | 사건 개요, 등장인물 명단, 기본 관계도, 맵 구조, NPC 존재와 위치 |
| **반공개** | 일부 플레이어 (AI 결정) | 특정 목격 정보, 공유된 비밀. 공유된 비밀이 동맹 형성의 씨앗이 됨 |
| **비공개** | 본인만 | 개인 목표, 개인 비밀, 특수 역할 여부 |

**예시:**
```
공개  → 모든 플레이어: "피해자는 자정에 사망했다. 용의자는 너희 5명이다."
반공개 → Alice와 Bob만: "두 사람은 어젯밤 피해자와 함께 있었다."
비공개 → Bob만: "당신이 범인이다. 들키지 마라."
```

이 정보 비대칭이 플레이어들이 서로 대화하고, 이동하고, 동맹을 맺고, 의심하게 만드는 핵심 동력이다.

```
┌─────────────────────────────────────────────────┐
│ [WOLF-7423] 저택의 밤 - 1층 거실               │
│ 여기: Alice, Bob  /  부엌: Carol  /  서재: Dave │
├─────────────────────────────────────────────────┤
│                                                 │
│  [GM] 갑자기 저택의 전등이 꺼집니다...           │
```

- 하단: 입력창 + 주요 커맨드 힌트

## 게임 종료 후: 저장 & 피드백

### 저장 데이터
게임이 끝나면 해당 세션의 모든 데이터를 저장한다:
- 생성된 스토리 전문 (세계관, 사건, 역할 배정 등)
- 맵 구조
- 모든 기믹 및 단서 배치
- NPC 퍼소나 및 설정
- 플레이어 행동 로그 (선택적)

### 피드백 수집
게임 종료 엔딩 화면 이후 간단한 피드백 수집:
- 스토리 재미도
- 몰입도
- 자유 텍스트 코멘트 (선택)

피드백은 저장된 스토리 데이터와 연결해 보관. 추후 스토리 생성 품질 개선에 활용.

## UI & 출력 형식 원칙

**모든 AI 출력은 구조화된 스키마를 따른다.** 자유 형식 텍스트 출력은 UI 파싱 오류로 직결되므로 금지.

AI가 생성하는 모든 이벤트는 미리 정의된 타입 중 하나여야 한다:

| 이벤트 타입 | 설명 |
|-------------|------|
| `narration` | GM 서술 텍스트 |
| `npc_dialogue` | NPC 발화 |
| `npc_give_item` | NPC가 플레이어에게 아이템 전달 |
| `npc_receive_item` | NPC가 플레이어로부터 아이템 수령 |
| `npc_reveal` | NPC가 증표/증거물 공개 (트리거 조건 충족 시) |
| `clue_found` | 단서 발견 |
| `story_event` | 스토리 분기 이벤트 발생 |
| `game_end` | 종료 조건 달성 |

UI는 이 타입 목록에 대응하는 렌더러를 갖추어야 한다. 새로운 인터랙션이 필요하면 타입을 추가하고 렌더러를 확장하는 방식으로 확장한다.

## 기술 결정 사항

- **기술 스택**: Go 1.26
- **배포 방식**: npm 패키지로 배포 (Go 바이너리를 npm으로 래핑, esbuild 패턴). `npx story host` / `npx story join` 명령어는 유지.
- **네트워크**: WebSocket (클라이언트-서버). 호스트가 WebSocket 서버 실행 (`github.com/gorilla/websocket` v1.5.3). 클라이언트 직접 연결.
- **API 키 관리**: 호스트가 본인의 API 키 사용. 환경변수 또는 로컬 설정 파일.
- **TUI**: Bubble Tea v2 (`charm.land/bubbletea/v2` v2.0.1) + Lip Gloss v2 (`charm.land/lipgloss/v2` v2.0.0) + Bubbles v2 (`charm.land/bubbles/v2` v2.0.0)
- **AI SDK**: 커스텀 AI Provider 인터페이스. 멀티 프로바이더 지원 (OpenAI `openai-go/v3` v3.24.0, Anthropic `anthropic-sdk-go` v1.26.0, Gemini `google.golang.org/genai` v1.49.0, Grok, DeepSeek). 세계 생성 시 여러 모델을 역할별로 오케스트레이션하여 스토리, 기믹, NPC 등을 병렬 생성
