## 1. 개요 (Overview)

### 프로젝트명

**Story** -- AI 구동 멀티플레이어 텍스트 RPG

### 한줄 정의

룸 코드 하나로 모인 친구들이, AI가 즉석에서 만든 세계에서 각자의 역할을 받고 채팅과 이동으로 스토리를 풀어나가는 터미널 소셜 RPG.

### 비전

누구나 터미널에서 `npx` 한 줄로 실행하고, 룸 코드 하나만 공유하면 친구들과 함께 AI가 생성한 고유한 세계에 몰입할 수 있는 경험을 만든다. 준비 시간 제로, 매번 새로운 스토리, 플레이어 간 대화와 이동이 곧 게임 메커닉이 되는 소셜 RPG를 목표로 한다.

### 목표 사용자 (타겟 유저)

| 우선순위 | 유저 그룹 | 특징 |
|----------|-----------|------|
| 1 (핵심) | 소셜 게이머 (친구 그룹) | 저녁 모임, 술자리, 파티에서 함께 즐기는 형태. 게임 규칙을 배울 필요 없이 바로 참여 가능해야 함 |
| 2 | 개발자 / 기술 친화적 유저 | 터미널 환경에 익숙하며, `npx` 실행에 거부감이 없는 사용자 |
| 3 | TRPG / 보드게임 팬 | 마피아, 머더미스터리, 방탈출 등 기존 소셜 디덕션 게임을 즐기는 유저 |
| 4 | 일반 캐주얼 게이머 | 친구의 초대로 처음 접하는 비기술 유저. 진입 과정이 최대한 단순해야 함 |

### 핵심 가치 제안 (Value Proposition)

1. **진입 장벽 제로**: `npx story host` 한 줄로 게임 생성. 별도 설치, 회원가입, 준비물 없음
2. **매번 새로운 세계**: AI가 장르, 배경, 사건, 역할을 매번 새롭게 생성하여 리플레이 가치가 높음
3. **정보 비대칭이 만드는 사회적 긴장감**: 공개/반공개/비공개 정보 레이어가 대화, 이동, 동맹, 의심을 자연스럽게 유발
4. **방(Room)이 곧 프라이버시**: 별도 밀담 기능 없이, 물리적 이동으로 프라이버시를 확보하는 직관적 구조
5. **터미널 네이티브**: 브라우저, 앱 없이 터미널에서 완결되는 경험

## 2. 목표 및 성공 지표 (Goals & Success Metrics)

### 비즈니스 목표

| ID | 목표 | 설명 |
|----|------|------|
| BG-1 | npm 패키지 공개 배포 | npmjs.com에 `story` 패키지를 배포하여 누구나 `npx story`로 실행 가능 |
| BG-2 | 오픈소스 커뮤니티 형성 | GitHub 리포지토리를 공개하고 컨트리뷰터를 확보 |
| BG-3 | 월간 활성 세션 수 확보 | 출시 3개월 내 월 500+ 게임 세션 달성 |
| BG-4 | 유저 피드백 기반 개선 루프 구축 | 게임 종료 후 피드백을 수집하고 AI 스토리 생성 품질을 지속 개선 |

### 사용자 경험 목표

| ID | 목표 | 설명 |
|----|------|------|
| UX-1 | 3분 내 게임 시작 | 호스트가 명령어 실행 후 모든 참가자가 합류하여 게임이 시작되기까지 3분 이내 |
| UX-2 | 설명서 없이 플레이 가능 | 게임 내 브리핑과 UI 힌트만으로 첫 플레이어도 바로 참여 가능 |

| UX-4 | 게임 후 회상 가능 | 엔딩에서 각 플레이어의 행동을 되돌아보는 개인화된 결말이 제공됨 |

| AI 응답 시간 (게임 중) | 3초 이내 (p95) | 서버 측 로그에서 측정 |

## 3. 사용자 스토리 & 시나리오 (User Stories)

### 3.1 호스트 사용자 시나리오

| ID | 사용자 스토리 |
|----|--------------|
| US-H01 | 호스트로서, `npx story host`를 실행하여 새 게임 세션을 생성하고 싶다. 그래야 친구들을 초대할 수 있다. |
| US-H02 | 호스트로서, 게임이 생성되면 룸 코드를 확인하고 친구들에게 공유하고 싶다. 그래야 모두가 같은 세션에 참여할 수 있다. |
| US-H03 | 호스트로서, 모든 참가자가 접속한 것을 확인한 후 게임을 시작하고 싶다. 그래야 누락 없이 모두 함께 시작할 수 있다. |
| US-H04 | 호스트로서, AI가 세계를 생성하는 동안 진행 상황을 확인하고 싶다. 그래야 대기 시간에 불안감이 없다. |
| US-H05 | 호스트로서, 게임 생성 시 플레이어 수에 맞는 적절한 맵과 역할이 자동으로 구성되기를 원한다. 그래야 별도의 설정 없이도 균형 잡힌 게임이 진행된다. |
| US-H06 | 호스트로서, AI API 키를 설정하고 싶다. 그래야 AI 기능을 사용할 수 있다. |

### 3.2 참가자 사용자 시나리오

| ID | 사용자 스토리 |
|----|--------------|
| US-J01 | 참가자로서, `npx story join <룸코드>`를 실행하여 친구가 만든 게임에 참가하고 싶다. 그래야 함께 플레이할 수 있다. |
| US-J02 | 참가자로서, 닉네임을 입력하고 대기실에서 다른 참가자들의 접속 현황을 확인하고 싶다. 그래야 누가 왔는지 알 수 있다. |
| US-J03 | 참가자로서, 게임 시작 시 공개 브리핑(세계관, 사건 개요, 등장인물 명단)을 받고 싶다. 그래야 게임 상황을 이해할 수 있다. |
| US-J04 | 참가자로서, 나만 볼 수 있는 역할, 개인 목표, 비공개 정보를 받고 싶다. 그래야 나의 역할에 몰입할 수 있다. |

### 3.3 게임 진행 중 시나리오

| ID | 사용자 스토리 |
|----|--------------|
| US-G01 | 플레이어로서, 현재 같은 방에 있는 사람들과 대화하고 싶다. 그래야 정보를 교환하고 추리를 진행할 수 있다. |
| US-G02 | 플레이어로서, 전체 플레이어에게 공개 발언을 하고 싶다. 그래야 위치와 관계없이 중요한 선언을 할 수 있다. |
| US-G03 | 플레이어로서, 맵의 다른 방으로 이동하고 싶다. 그래야 특정 플레이어와 밀담하거나 새로운 단서를 찾을 수 있다. |
| US-G04 | 플레이어로서, `/examine` 명령으로 현재 방의 물건이나 환경을 조사하고 싶다. 그래야 숨겨진 단서를 발견할 수 있다. |
| US-G05 | 플레이어로서, `/do` 명령으로 행동을 서술하고 싶다. 그래야 롤플레이를 통해 스토리에 기여할 수 있다. |
| US-G06 | 플레이어로서, 같은 방에 있는 NPC에게 말을 걸고 싶다. 그래야 NPC가 가진 정보를 얻을 수 있다. |
| US-G07 | 플레이어로서, 현재 맵 전체의 구조와 각 방에 누가 있는지 확인하고 싶다. 그래야 이동을 계획할 수 있다. |
| US-G08 | 플레이어로서, 내가 획득한 단서와 아이템 목록을 확인하고 싶다. 그래야 현재 상황을 정리할 수 있다. |
| US-G09 | 플레이어로서, 내 역할과 개인 목표를 다시 확인하고 싶다. 그래야 잊어버린 정보를 되살릴 수 있다. |
| US-G10 | 플레이어로서, GM의 스토리 이벤트나 서술을 받고 싶다. 그래야 게임의 긴장감이 유지되고 스토리가 진전된다. |
| US-G12 | 플레이어로서, NPC로부터 아이템을 받거나, NPC에게 아이템을 전달하고 싶다. 그래야 스토리를 진행시킬 수 있다. |
| US-G13 | 플레이어로서, 도움말(`/help`)을 확인하고 싶다. 그래야 사용 가능한 명령어를 알 수 있다. |

### 3.4 게임 종료 시나리오

| ID | 사용자 스토리 |
|----|--------------|
| US-E01 | 플레이어로서, 게임의 종료 조건이 달성되었을 때 AI가 결과를 판정하고 공개하는 것을 보고 싶다. 그래야 게임의 성패를 알 수 있다. |
| US-E02 | 플레이어로서, 게임 종료 시 AI가 내 행동을 요약한 개인화된 결말을 보고 싶다. 그래야 게임을 되돌아보는 재미가 있다. |
| US-E03 | 플레이어로서, 게임 종료 후 모든 비밀 정보가 공개되는 것을 보고 싶다. 그래야 "아, 그때 그게 그런 거였구나"하는 카타르시스를 느낄 수 있다. |
| US-E04 | 플레이어로서, 게임 종료 후 간단한 피드백(재미도, 몰입도)을 남기고 싶다. 그래야 서비스 개선에 기여할 수 있다. |
| US-E05 | 플레이어로서, 게임 종료 후 해당 세션의 스토리 데이터가 저장되기를 원한다. 그래야 나중에 다시 돌아볼 수 있다. |
| US-E06 | 플레이어로서, 시간 초과로 게임이 끝날 때 AI가 자연스러운 결말을 서술해 주기를 원한다. 그래야 미완결 느낌 없이 마무리된다. |
| US-E07 | 플레이어로서, 원할 때 `/end` 명령으로 게임 종료를 제안하고 싶다. 그래야 충분히 즐겼거나 시간이 부족할 때 자연스럽게 마무리할 수 있다. |

### 4.2 AI 세계 생성 시스템

| ID | 설명 | 수용 조건 | 우선순위 | 의존성 |
|----|------|-----------|----------|--------|

## 6. AI 시스템 설계 요구사항

### 6.1 AI 입출력 스키마 정의 방향

모든 AI 입출력은 **구조화된 JSON 스키마**를 따른다. 자유 형식 텍스트 출력은 UI 파싱 오류로 직결되므로 금지한다.

**세계 생성 출력 스키마 (개념 수준)**:

```
WorldGeneration {
  meta: {
    theme: string
    setting: string
    estimatedDuration: number (분, 10~30)
    hasGM: boolean
    hasNPC: boolean
  }
  world: {
    title: string
    synopsis: string
    atmosphere: string
  }
  gameStructure: {
    concept: string
    coreConflict: string
    progressionStyle: string
    commonGoal: string | null
    endConditions: EndCondition[]      // 복수 (fallback 포함)
    winConditions: WinCondition[]
    requiredSystems: ('vote' | 'consensus' | 'ai_judge')[]
    briefingText: string
  }
  map: {
    rooms: Room[]
    connections: Connection[]
  }
  characters: {
    playerRoles: PlayerRole[]          // players → playerRoles
    npcs: NPC[]
  }
  information: {
    public: PublicInfo
    semiPublic: SemiPublicInfo[]
    private: PrivateInfo[]
  }
  clues: Clue[]
  gimmicks: Gimmick[]
}
```

> 정확한 Go 구조체 정의는 `system-design/shared/schemas.md` 참조.

**게임 중 AI 응답 스키마 (개념 수준)**:

```
GameResponse {
  events: GameEvent[]
  stateChanges: StateChange[]
}

GameEvent {
  type: EventType
  target: "all" | "room" | "player"
  targetIds: string[]
  data: EventData  // 타입별 상이
}
```

### 6.2 이벤트 타입 시스템

concept 문서에 정의된 이벤트 타입을 기반으로 확장한다.

| 이벤트 타입 | 설명 | 대상 | 데이터 |
|-------------|------|------|--------|
| `narration` | GM 서술 텍스트 | 전체 또는 특정 방 | `{ text: string, mood: string }` |
| `npc_dialogue` | NPC 발화 | 같은 방 | `{ npcId: string, npcName: string, text: string, emotion: string }` |
| `npc_give_item` | NPC가 플레이어에게 아이템 전달 | 같은 방 | `{ npcId: string, playerId: string, item: Item }` |
| `npc_receive_item` | NPC가 플레이어로부터 아이템 수령 | 같은 방 | `{ npcId: string, playerId: string, item: Item }` |
| `npc_reveal` | NPC가 증표/증거물 공개 | 같은 방 | `{ npcId: string, revelation: string, clue: Clue }` |
| `clue_found` | 단서 발견 | 같은 방 | `{ playerId: string, clue: Clue, location: string }` |
| `story_event` | 스토리 분기 이벤트 | 전체 또는 특정 방 | `{ title: string, description: string, consequences: string[] }` |
| `game_end` | 종료 조건 달성 | 전체 | `{ reason: string, ending: Ending }` |
| `examine_result` | 조사 결과 | 같은 방 | `{ playerId: string, target: string, description: string, clueFound: boolean }` |
| `action_result` | 행동(/do) 결과 | 같은 방 | `{ playerId: string, action: string, result: string, triggeredEvents: string[] }` |
| `player_move` | 플레이어 이동 | 전체 | `{ playerId: string, from: string, to: string }` |
| `time_warning` | 시간 경고 | 전체 | `{ remainingMinutes: number }` |

> 투표(`vote_started`, `vote_progress`, `vote_ended`)는 GameEvent가 아닌 ServerMessage로 처리.
> system-design에서 투표는 시스템 레벨 조작으로 분류. `shared/protocol.md` 참조.

### 6.3 프롬프트 설계 원칙

| 원칙 | 설명 |
|------|------|
| **자유 생성, 구조화 출력** | 스토리 생성 프롬프트는 의도적으로 모호하고 열려있게 설계하되, 출력은 엄격한 JSON 스키마를 따르도록 한다 |
| **재미 최우선** | 모든 프롬프트에 "재미있고 몰입되는 스토리가 유일한 기준"임을 명시한다 |
| **개연성보다 흥미** | 현실적 개연성보다 게임으로서의 흥미를 우선하도록 지시한다 |
| **정보 비대칭 강조** | 공개/반공개/비공개 레이어의 균형을 반드시 고려하도록 프롬프트에 포함한다 |
| **플레이어 수 적응** | 플레이어 수에 따라 역할 복잡도, 맵 크기, 단서 수를 조절하도록 지시한다 |
| **컨텍스트 유지** | 게임 중 AI 호출 시 이전 대화 맥락, 플레이어 행동 히스토리, 현재 게임 상태를 포함한다 |
| **응답 제한** | 단일 응답의 최대 길이를 제한하여 과도한 서술을 방지한다 |
| **페르소나 일관성** | NPC 대화 프롬프트에 해당 NPC의 전체 퍼소나 정보를 포함하여 일관성을 보장한다 |

### 6.4 AI 응답 품질 기준

| 기준 | 설명 | 측정 방법 |
|------|------|-----------|
| **스키마 준수율** | AI 응답이 정의된 JSON 스키마를 따르는 비율 | 파싱 성공률 (목표: 95% 이상) |
| **NPC 일관성** | NPC가 설정된 퍼소나를 유지하는 비율 | 수동 검수 및 사용자 피드백 |
| **서술 품질** | GM 서술과 엔딩의 문학적 품질 | 사용자 피드백 (재미도/몰입도 평점) |
| **정보 보호** | AI가 비공개 정보를 다른 플레이어에게 누출하지 않는 비율 | 자동화된 정보 누출 감지 |
| **맥락 유지** | 이전 대화와 행동을 반영한 응답 비율 | 수동 검수 |
| **응답 다양성** | 비슷한 입력에 대해 다양한 응답을 생성하는 정도 | 반복 테스트 |

### 7.3 정보 가시성(Visibility) 모델

정보의 가시성은 세 레이어로 관리된다. 서버에서 모든 정보를 관리하며, 클라이언트에게는 해당 플레이어가 볼 수 있는 정보만 전송한다.

| 데이터 | 가시성 규칙 |
|--------|-------------|
| 맵 구조 (방 이름, 연결 관계) | 전체 공개 |
| 각 방에 있는 플레이어 | 전체 공개 |
| 각 방에 있는 NPC | 전체 공개 |
| 방 상세 설명 | 해당 방에 있는 플레이어만 |
| 같은 방 채팅 메시지 | 같은 방 플레이어만 |
| 글로벌 채팅 메시지 | 전체 공개 |
| NPC 대화 내용 | 같은 방 플레이어만 |
| /examine 결과 | 같은 방 플레이어만 |
| /do 결과 | 같은 방 플레이어만 |
| GM 서술 | 대상에 따라 (전체 또는 특정 방) |
| 플레이어 역할 (본인) | 본인만 |
| 플레이어 역할 (타인) | 비공개 (종료 후 공개) |
| 개인 목표 | 본인만 |
| 비밀 정보 | 본인만 (반공개는 지정된 그룹) |
| 인벤토리 | 본인만 |
| 발견한 단서 목록 | 본인만 |
| 이동 사실 | 전체 공개 |

**서버 측 필터링 원칙**: 클라이언트를 신뢰하지 않는다. 모든 가시성 필터링은 서버에서 수행하며, 클라이언트에게는 볼 수 있는 정보만 전송한다.

### 9.3 MVP 완료 기준

MVP는 다음 조건을 모두 충족해야 완료된 것으로 간주한다.

1. **실행 가능성**: `npx story host`로 게임을 생성하고, `npx story join <코드>`로 참가할 수 있다
2. **세계 생성**: AI가 세계, 맵, 역할, 단서를 생성하고 검증을 통과한다
3. **게임 진행**: 채팅, 이동, examine, do, NPC 대화, 투표가 동작한다
4. **정보 비대칭**: 공개/반공개/비공개 정보가 올바른 대상에게만 전달된다
5. **게임 종료**: AI가 설계한 종료 조건 판정, 시간 초과 종료, 플레이어 주도 종료(`/end`)가 동작한다
6. **엔딩**: 개인화된 엔딩 서술과 전체 비밀 공개가 제공된다
7. **안정성**: 3~6명 플레이어로 60분간 크래시 없이 게임이 진행된다
8. **플레이테스트**: 최소 5회 이상의 완전한 게임 세션이 성공적으로 진행된다

## 10. 리스크 & 완화 전략

### 10.1 기술적 리스크

| 리스크 | 심각도 | 발생 가능성 | 완화 전략 |
|--------|--------|-------------|-----------|
| **방화벽/네트워크로 인한 연결 실패** | 중간 | 중간 | 1) 같은 네트워크에서는 자동 연결 2) 원격 시 포트 포워딩 안내 3) 향후 릴레이 서버로 100% 해결 가능 |
| **AI 응답이 JSON 스키마를 따르지 않음** | 높음 | 중간 | 1) structured output / function calling 활용 2) JSON 파싱 실패 시 최대 3회 재시도 3) 재시도 실패 시 fallback 응답 제공 |
| **AI 응답 지연 (>3초)** | 중간 | 중간 | 1) 스트리밍 응답으로 체감 지연 최소화 2) 대기 중 시각적 피드백 제공 3) 타임아웃 설정 및 재시도 로직 |
| **호스트 프로세스 크래시** | 높음 | 낮음 | 1) 에러 핸들링 강화 2) graceful shutdown 구현 3) 향후 상태 스냅샷/복구 기능 검토 |
| **WebSocket 연결 불안정** | 중간 | 중간 | 1) WebSocket 상태 모니터링 2) 자동 재연결 (최대 3회) 3) 재연결 시 상태 복원 |

## 부록 A: 용어 정의

| 용어 | 정의 |
|------|------|
| 호스트(Host) | `npx story host`를 실행하여 게임 세션을 생성하는 플레이어. 서버 역할도 겸함 |
| 참가자(Joiner) | `npx story join <코드>`로 세션에 참가하는 플레이어 |
| 룸 코드(Room Code) | 게임 세션 식별을 위한 고유 코드 (예: WOLF-7423) |
| 방(Room) | 맵 내의 물리적 공간. 프라이버시의 기본 단위 |
| GM (Game Master) | AI가 운영하는 게임 진행자. 스토리에 따라 있을 수도 없을 수도 있음 |
| NPC (Non-Player Character) | AI가 운영하는 비플레이어 캐릭터 |
| 정보 레이어 | 공개/반공개/비공개로 구분되는 정보 분류 체계 |
| 이벤트 타입 | AI 출력의 구조화된 분류 (narration, clue_found 등) |
| 기믹(Gimmick) | 특정 조건에서 발동하는 숨겨진 게임 장치 |
| 단서(Clue) | 플레이어가 발견할 수 있는 정보 조각 |
| 게임 구조(Game Structure) | AI가 세계 생성 시 설계하는 게임의 핵심 갈등, 진행 방식, 종료 조건, 승리 판정 기준의 총체. 사전 정의된 모드 분류 없이 매번 자유롭게 설계됨 |
